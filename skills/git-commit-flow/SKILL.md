# git-commit-flow

Commits content files to git with a conventional message format.
Enables `git-diff-analyzer` to find and compare AI output vs. human corrections, 
so that the agent skills can learn from these corrections.

## What Counts as Content

Any human-readable content generated by the agent — everything except program code. Examples:

- Articles, blog posts, newsletters
- Reports, analyses, executive summaries
- Research notes, literature reviews
- Proposals, SOWs, business plans
- Documentation, guides, how-to manuals
- Presentations and slide decks (text)
- Social media posts, marketing copy

**Excluded**: program code, configuration files; skills, rules and other context files.

## Workflow

### Step 1: Analyze eligible changes

```bash
git status --short
```

Do nothing and exit if any of the following is true:

- **No changes at all** — working tree is clean.
- **No content files changed** — all changes are in program code, configuration,
  skill files, or rule files (see *What Counts as Content* above).

Ask the user before proceeding if the modified files are scattered across multiple UNRELATED folders.
In this case, propose two or more commits.

> [!NOTE]
> If a few RELATED folders contain the modified files, do multiple commits (one per folder)
> without asking the user.
> Subfolders of the same folder are related, but subfolders of the project root folder aren't.

> [!NOTE]
> If modified files exist both in a parent folder and its subfolder(s), do a single commit 
> linked to the parent folder.

### Step 2: Determine the commit type

Two dimensions:

| Dimension | Options |
|-----------|---------|
| **Action** | `feat` (new file created) · `fix` (existing file updated) |
| **Author** | `ai` (agent just generated this) · `human` (user edited agent output) |

Decision logic:
- If triggered **right after the agent generated content** → author is `ai`
- If triggered **because the user signaled they edited AI output** (e.g., "I
  fixed it", "check my edits", "commit my changes") → author is `human`
- If the origin of the changes is unclear → authors are both: `human/ai`
  (although this must not happen if CLAUDE.md / AGENTS.md file include proper instructions).
- If **new content files** were created → `feat`
- If **existing content files** were modified → `fix`
- If both new and modified → use `feat` for the commit message

> [!IMPORTANT]
> If both `feat` and `human` apply, ask the user whether a commit is needed,
> and if yes, what it is about (to generate the right commit message).
> If yes, use `feat(<folder name>): human` as the prefix at the next stage.

### Step 3: Stage and commit

Stage only the content files that changed (do not stage unrelated changes):

```bash
git add <changed content files>
git commit -m "<message>"
```

**Message formats:**

| Situation | Format |
|-----------|--------|
| Agent updated existing content | `fix(<folder name>): ai <verb> <files touched, or sections if single file>` |
| Agent created new content file(s) | `feat(<folder name>): ai <verb> <files created>` |
| User edited content | `fix(<folder name>): human <verb> <files touched, or sections if single file>` |

> [!NOTE]
> The verb must be in the past form (e.g., `enhanced`, `created`, `corrected`, `fixed`).

> [!IMPORTANT]
> The description names **where changes happened** — which files, or which sections of a single file.
> Never summarize what changed in the content. This is what enables `git-diff-analyzer` to match commit pairs 
> by location and avoid analyzing **what exactly changed** before git diff.

To write the description in `human/ai` case (unclear origin), inspect the git working tree (`git diff`)
to see exactly which files (or file sections if there's only one file) were touched

> [!NOTE]
> If the user prompt mentions what commit message should include, use it as a priority; but keep `feat(<folder name>): <author>` prefix anyway.

**Message examples:**

- `feat(202-managing-agents): ai created the article sections from user brain dump`
- `fix(202-managing-agents): ai enhanced intro section`
- `fix(202-managing-agents): human shortened conclusion and made edits throughout the article`
- `fix(202-managing-agents): ai enhanced title and subtitle`
- `fix(202-managing-agents): ai enhanced language of the article`
- `fix(202-managing-agents): human corrected subtitle`

> [!NOTE]
> `git-diff-analyzer` matches commit pairs by **location overlap** — same folder scope and overlapping
> files or sections in the description. Commit B (human) might not be the next commit after Commit A (ai).
> Example pair above: `ai enhanced title and subtitle` ↔ `human corrected subtitle`.

### Step 4: Confirm

Report the commit hash and message to the user.
